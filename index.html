1. Система флагов (path_walk, path_look) - LOGIC SDK 0.4

Статья 1 | Логика сталкеров SDK 0.4

В точках путей можно задавать флаги, изменяющие поведение персонажа. Флаги задаются прямо в имени waypoint-а, например, для точки с именем "wp00": wp00|flag1|flag2.

3.1. Флаги точек пути path_walk:

a=state

Выбирает состояние тела при перемещении (Только из раздела – Ходячие состояния). Список состояний можно взять в gamedata\scripts\state_lib.script.

p=percent

Вероятность остановиться в точке в процентах (0 – 100). По умолчанию 100, т.е. сталкер никогда не проходит мимо точек остановки.

sig=name

Установить сигнал с именем name сразу по прибытию в точку (до поворота) для последующей его проверки с помощью поля on_signal логической схемы. Если нужно установить сигнал после поворота – используйте соответствующий флажок пути path_look.

Флаги точек пути path_look:

a=state

Выбирает состояние тела при стоянии (или сидении) на месте. (Из разделов Стоячие и Сидячие состояния) Список состояний можно взять в gamedata\scripts\state_lib.script.

t=msec

Время в миллисекундах, которое персонаж должен смотреть в заданную точку. '•' – бесконечное время. Допустимы значения в диапазоне [1000, 30000], по умолчанию – 5000. Для конечных (терминальных) вершин пути path_walk, у которых не более 1-й соответствующей точки path_look, значение t всегда считается бесконечным и его явно задавать не нужно.

sig=name

После поворота в точку path_look, установить сигнал с именем name.

syn

Наличие флажка задержит установку сигнала до тех пор, пока в точку с флажком syn не прибудут все персонажи с данным team-ом (team задается в виде текстовой строки в customdata). До тех пор, пока остальные персонажи не прибудут, ожидающей персонаж будет отыгрывать свою idle анимацию.

sigtm=signal

Устанавливает сигнал при вызове time_callback-а state manager-ом. Соответственно, если t=0, то сигнал будет установен после отыгрывания init анимации. Это используется, например, с анимацией press, которая состоит из двух частей: 1 - нажимаем на кнопку, 2 - опускаем рука.

В пути path_look можно сделать: wp00 | a=press | t=0 | sigtm=pressed
А затем переключить схему: on_signal = pressed | другая_схема

3.1.1. Более подробное описание путей (walker)

На карту для каждого walker'а нужно поставить:

  1. Путь path_walk, по которому walker ходит.
  2. Путь path_look, состоящим из точек, в которые walker смотрит.

Walker'ов может быть от одного или больше. Они могут действовать независимо, или взаимодействовать друг с другом.

[walker]
team = ... - имя команды, произвольная текстовая строка
path_walk = ... - имя пути
path_look = ... - (не обязательно) имя пути

Если персонаж должен только ходить по маршруту, path_look можно не задавать. Если персонаж должен стоять на месте, то ему задается одна точка пути path_walk и как минимум одна точка пути path_look.

Пример настройки пути в LOGIC SDK 0.4:

; Пример waypoint с флагами в LOGIC SDK 0.4
[wp00]:walker_path
a = patrol
p = 100
sig = reached_point

[wp01]:walker_path  
a = raid
p = 50
sig = check_area

2. Схемы поведения сталкеров - LOGIC SDK 0.4

Статья 2 | Поведение NPC SDK 0.4

Основные схемы поведения сталкеров:

walker

Базовая схема перемещения по патрульным путям. Используется для патрулирования местности.

remark

Схема для синхронизации/связки других схем. Позволяет настроить диалоги и взаимодействия.

sleeper

Схема для NPC, которые спят или отдыхают. Просыпаются при приближении игрока или при сигнале.

kamp

Схема для обороны позиции. NPC остаются на месте и защищают территорию.

camper

Схема засады. NPC прячутся и ждут приближения врага.

sniper

Схема снайпера. NPC занимают позицию и ведут огонь с дальнего расстояния.

follower

Схема следования. NPC следуют за другим персонажем или игроком.

zoneguard

Схема охраны зоны. NPC патрулируют и охраняют определенную территорию.

wounded

Схема для раненых NPC. Персонажи с этой схемой ищут укрытие и пытаются выжить.

rest

Схема отдыха. NPC отдыхают, едят или занимаются другими повседневными делами.

heli_hunter

Специальная схема для охоты на вертолеты.

patrol

Схема патрулирования с расширенными настройками маршрутов.

3. Схемы walker и remark - LOGIC SDK 0.4

Статья 3 | Основные схемы SDK 0.4

3.1. Схема walker

Базовая схема перемещения NPC по патрульным путям. Настраивается через:

[walker]
path_walk = путь_патрулирования
path_look = путь_обзора
team = название_команды

Основные параметры:

  • path_walk - обязательный параметр, путь перемещения
  • path_look - необязательный, точки обзора
  • team - название команды для синхронизации
  • def_state_moving - состояние при движении
  • def_state_standing - состояние при остановке

3.2. Схема remark

Схема для диалогов и синхронизации действий NPC.

[remark]
anim = анимация_диалога
target = цель_для_диалога
sound = звук_диалога
synch = true/false

Пример синхронизированного диалога:

[logic]
active = remark
[remark]
anim = hello
target = actor
sound = dialogs\hello.ogg
synch = true
on_signal = end_remark| следующая_схема

4. Секции combat, death, hit - LOGIC SDK 0.4

Статья 4 | Боевые схемы SDK 0.4

4.1. Секция combat

Определяет поведение NPC в бою.

[combat]
enemy = цель_атаки
action = тип_боевого_поведения
on_info = {+инфопоршн} схема_активации

Типы боевого поведения:

  • hide - укрытие и стрельба из укрытия
  • sniper - снайперская стрельба
  • assault - штурм с приближением к врагу
  • retreat - отступление при низком здоровье

4.2. Секция death

Реакция на смерть NPC.

[death]
on_info = {+инфопоршн} схема_смерти
cond = условие_смерти

4.3. Секция hit

Реакция на получение урона.

[hit]
on_info = {+инфопоршн} схема_реакции
condlist = список_условий

5. Новые секции и danger - LOGIC SDK 0.4

Статья 5 | Дополнительные секции SDK 0.4

5.1. Новые секции SDK 0.4

dont_spawn_character_supplies

Запрещает спавн стандартных припасов на персонаже:

[dont_spawn_character_supplies]

no_smart

Отключает "умное" поведение NPC:

[no_smart]

threshold

Устанавливает пороговые значения для смены состояний:

[threshold]
min_health = минимальное_здоровье
max_health = максимальное_здоровье

5.2. Секция danger

Реакция на опасности в окружающей среде.

[danger]
ignore_distance = дистанция_игнорирования
type = тип_опасности
on_info = {+инфопоршн} схема_реакции

Типы опасностей:

  • grenade - граната
  • sound - подозрительный звук
  • hit - попадание
  • attack - атака
  • corpse - труп

6. Схемы для монстров - LOGIC SDK 0.4

Статья 6 | Монстры SDK 0.4

6.1. Основные схемы поведения монстров

mob_walker

Перемещение монстра по точкам пути:

[mob_walker]
path_walk = путь_движения
path_look = путь_обзора
home_min = минимальное_время_дома
home_max = максимальное_время_дома

mob_eluder

Монстр уклоняется от противника:

[mob_eluder]
min_radius = минимальная_дистанция
max_radius = максимальная_дистанция

mob_remark

Монстр делает паузу в определенной точке:

[mob_remark]
anim = анимация
time = время_паузы

mob_combat

Боевое поведение монстра:

[mob_combat]
min_attack_distance = минимальная_дистанция_атаки
max_attack_distance = максимальная_дистанция_атаки
home_path = путь_к_дому

mob_death

Реакция на смерть монстра:

[mob_death]
on_info = {+инфопоршн} схема_смерти

mob_jump

Прыжок монстра:

[mob_jump]
offset = смещение_прыжка
velocity = скорость_прыжка
anim = анимация_прыжка

mob_camp

Монстр стоит на месте и охраняет территорию:

[mob_camp]
path_look = точки_обзора
home_path = путь_к_дому

mob_home

Монстр возвращается домой:

[mob_home]
home_min = минимальное_время_дома
home_max = максимальное_время_дома

mob_fake_death

Монстр притворяется мертвым:

[mob_fake_death]
anim = анимация_смерти
on_info = {+инфопоршн} схема_оживления

7. Байки из склепа и спавнеры - LOGIC SDK 0.4

Статья 7 | Истории и спавн SDK 0.4

7.1. Истории у костра

Настройка диалогов у костра с использованием схемы remark:

[logic]
active = campfire_story
[campfire_story]
path_look = campfire_look
path_walk = campfire_walk
story = название_истории
team = storytelling

Пример истории:

[campfire_story]
anim = sit_ass
sound = stories\campfire\story_1.ogg
on_signal = end_story| следующая_схема

7.2. Секция dont_spawn_loot

Запрещает спавн лута на NPC после смерти:

[dont_spawn_loot]

7.3. Секция spawner

Спавн NPC и монстров по условиям:

[spawner]
cond = условие_спавна
npc = тип_NPC
max_count = максимальное_количество
spawn_radius = радиус_спавна

Пример дневных монстров:

[spawner]
cond = {=time_of_day day}
npc = boar
max_count = 3
spawn_radius = 50

Пример ночных монстров:

[spawner]
cond = {=time_of_day night}
npc = bloodsucker
max_count = 2
spawn_radius = 30

8. Оверрайды и скрипт logic - LOGIC SDK 0.4

Статья 8 | Системные настройки SDK 0.4

8.1. Оверрайды для сталкеров

Переопределение стандартного поведения NPC:

[overrides]
combat_ignore = всегда_игнорировать_бой
combat_ignore_keep_when_attacked = игнорировать_даже_при_атаке

Пример оверрайда:

[overrides]
combat_ignore = {=fighting_dist_ge(20)} true, false
combat_type = {=best_enemy} monolith

8.2. Оверрайды для монстров

[overrides]
active = всегда_активный
predator = режим_хищника

8.3. Скрипт logic

Основной скрипт для управления переключением схем:

[logic]
active = начальная_схема
on_actor_dist_le = дистанция_актора| схема_активации
on_info = {+инфопоршн} схема_активации

Пример логики с условиями:

[logic]
active = walker
on_actor_dist_le = 10| remark@hello
on_info = {+alarm} combat
on_info = {-alarm} walker

Условия переключения схем:

  • on_actor_dist_le - дистанция до актора меньше или равна
  • on_actor_dist_ge - дистанция до актора больше или равна
  • on_signal - при получении сигнала
  • on_info - при изменении инфопоршна
  • on_timer - по таймеру

8.4. Работа с кастомными данными

[custom_data]
key1 = значение1
key2 = значение2
on_info = {+ключ} действие

9. Синтаксис скрипта Logic - LOGIC SDK 0.4

Статья 9 | Синтаксис SDK 0.4

9.1. Основные элементы синтаксиса

Секции и параметры:

[section_name]
parameter1 = value1
parameter2 = value2

Комментарии:

; Это комментарий
parameter = value ; комментарий после значения

9.2. Условия

Простые условия:

{=condition} value_if_true, value_if_false

Сложные условия:

{=condition1 =condition2} value

Логические операторы:

  • = - равно
  • != - не равно
  • < - меньше
  • > - больше
  • <= - меньше или равно
  • >= - больше или равно
  • ! - отрицание
  • & - И
  • | - ИЛИ

9.3. Эффекты

Установка инфопоршнов:

[logic]
on_info = {+info_portshion} next_scheme

Вызов функций:

[logic]
on_signal = signal_name| function_call

9.4. Переключение секций

Простое переключение:

[logic]
active = first_scheme
on_info = {+trigger} second_scheme

Переключение с параметрами:

[logic]
active = walker@path1
on_info = {+alarm} combat@aggressive

9.5. Работа с инфопоршнами

Проверка инфопоршнов:

{+info_portshion_1 -info_portshion_2} scheme_name

Установка/снятие инфопоршнов:

[scheme]
on_info = {+set_info_portshion} next_scheme
on_info = {-clear_info_portshion} another_scheme

9.6. Сложные логические конструкции

Вложенные условия:

{=check_dist_le(10) =has_enemy} combat
{=check_dist_le(10) !has_enemy} remark@hello

Цепочки условий:

[logic]
active = walker
on_actor_dist_le = 5| {=time_of_day day} remark@day_greeting, remark@night_greeting
on_info = {+quest_started} follower

10. Схемы работы объектов и смарттеррейны - LOGIC SDK 0.4

Статья 10 | Объекты и смарттеррейны SDK 0.4

10.1. Прожекторы (searchlights)

Базовые настройки:

[searchlight]
path = путь_движения
color = цвет_луча
range = дальность_освещения

Пример прожектора с трекингом:

[searchlight]
path = patrol_path
color = yellow
range = 100
target = actor ; слежение за игроком

10.2. Кодовые замки (code_lock)

Что такое GUI интерфейсы

GUI интерфейсы в сфере моддинга на сталкер - это активные окна, запускаемые из скриптов. Внутри которых могут быть:

  1. Кнопки
  2. Статичные изображения
  3. Листы
  4. Текстовые поля
  5. Текст
  6. Чекбоксы
  7. Вкладки
  8. Рамки
  9. Фреймлайны
  10. Спин намы
  11. Группы параметров

GUI интерфейсы состоят из XML файла разметки и скрипта класса. Такие классы как классы инвентаря и ПДА зашиты в движок, и напрямую редактировать их без исходников не выйдет.
